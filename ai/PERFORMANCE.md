## Current Performance Profile

### Backend

| Concern | Approach | Notes |
|---------|---------|-------|
| Trade lookup by user | `Trade.find({ assignedTo: userId }).lean()` | Uses `assignedTo` index |
| Queue trade batch fetch | `Trade.find({ tradeRef: { $in: refs } }).lean()` | Single batched `$in` query |
| Reply processing | In-memory `_cachedTrades` lookup (no DB hit) | Refreshed every 2s |
| Audit logging | Fire-and-forget `.catch()` — never blocks response | Non-critical path |
| Socket event emit | Immediate after DB write — same process, <1ms | No external hop |
| Trade action DB write | Targeted `Trade.updateOne({ tradeRef }, { $set: {...} })` | Updates specific fields only |

| Concern | Approach | Notes |
|---------|---------|-------|
| Queue updates | Socket.io `trade_update` event (primary) | Low latency real-time |

## Open Performance Issues

### KP-001 | N+1 risk in `/api/conversation/shared`
**File**: `src/routes/conversationRoutes.js`
**Description**: For each active queue found, conversations are fetched per tradeRef individually — potential N+1 query pattern when many sessions are active.
**Impact**: `O(n)` DB queries where `n = number of active sessions × trades per session`. At 10 concurrent users: ~200 individual queries per shared inbox load.
**Fix**:
```js
// Current (N+1 risk):
for (const tradeRef of allTradeRefs) {
  const conv = await conversationEngine.getConversation(tradeRef);
  // ...
}

// Fixed (single batched query):
const conversations = await Conversation.find({ tradeRef: { $in: allTradeRefs } });
```

### KP-002 | No pagination on `GET /api/trade/all`
**File**: `src/routes/tradeRoutes.js`
**Description**: Returns all trade documents with no `skip`/`limit`. As the pool grows (target: 1000+ trades), this returns megabytes of data on every request.
**Impact**: Admin/debug endpoint only (not used in normal training flow) but still dangerous at scale.
**Fix**: Add `?page=1&limit=50` query params with skip/limit logic.

### KP-003 | Large communication page (850 lines, ~40KB parsed JS)
**File**: `frontend/src/app/communication/page.js`
**Description**: Single 850-line React component. Slow to parse, slow initial render, difficult to code-split.
**Impact**: Adds to initial page load time. All sub-functionality (inbox, thread, compose, FO channel) must load before anything renders.
**Fix**: Split into `InboxList`, `MessageThread`, `ComposeModal`, `ReplyModal`, `FolderNav` sub-components with lazy loading.

### KP-004 | In-memory trade cache full rebuild every 2s
**File**: `server.js` (cache refresh `setInterval`)
**Description**:
```js
setInterval(async () => {
  const assignedTrades = await Trade.find({ assignedTo: { $ne: null } }).lean();
  communicationEngine._cachedTrades = {};
  assignedTrades.forEach(t => { communicationEngine._cachedTrades[t.tradeRef] = t; });
}, 2000);
```
This executes a full collection scan every 2 seconds regardless of whether anything changed.
**Impact**: At 100 concurrent users (2000 assigned trades), this loads ~2000 full trade documents every 2 seconds.
**Fix**: Use targeted cache invalidation — update `_cachedTrades[tradeRef]` on each trade action instead of full rebuild. Or use a TTL cache with selective invalidation.

### KP-005 | ~~`JSON.stringify` comparison for trade sync~~ ✅ Fixed
**Status**: Resolved. Workstation `useEffect` now compares `currentStatus` and `pendingAmendments.length` instead of full JSON serialization.

## Socket.io Performance Notes

- `trade_update` and `new_email` are currently emitted with `io.emit()` in some code paths — should be room-targeted to `desk_<desk>` or `user_<userId>` to avoid unnecessary client processing
- Each connected user joins 2 rooms (minimal overhead for expected user counts <100)
- Socket.io long-polling fallback is available but not explicitly configured; WebSocket upgrade is the default

## LLM Response Latency

- CPTY/FO replies are intentionally delayed 4–12 seconds to feel realistic
- Gemini API call latency: typically 1–3 seconds for the prompt sizes used
- Cerebras fallback latency: typically <1 second (faster inference)
- Offline static response: <1ms
- The delay is implemented as `setTimeout` of a random value within the `communicationEngine.scheduleCptyReply()` call — the LLM call itself happens at scheduling time, not at delivery time
