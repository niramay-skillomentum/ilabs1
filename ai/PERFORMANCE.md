# Performance

## Current Performance Characteristics

### Backend

| Concern | Current Approach | Notes |
|---------|-----------------|-------|
| Trade lookup | `Trade.find({ assignedTo: userId }).lean()` | Uses index on `assignedTo` |
| Queue trades | Array of `tradeRef` strings in `Queue.trades` | Bulk fetch via `$in` query |
| Cache | `_cachedTrades` in-memory object, refreshed every 2s | Avoids DB hit in reply processors |
| Reply processing | `setInterval` every 3s, single item per tick | Non-blocking, processes one reply at a time |
| Audit logging | Fire-and-forget `.catch()` | Never blocks response |
| Socket events | Emitted after DB write | In-process, low latency |
| Conversation bulk queries | `Trade.find({ tradeRef: { $in: refs } })` | Batched, not N+1 |

### Frontend

| Concern | Current Approach | Notes |
|---------|-----------------|-------|
| Queue polling | Background `setInterval` every 15s | Fallback only — Socket.io is primary |
| Selected trade sync | `useEffect` with JSON.stringify comparison | Prevents re-renders on unchanged data |
| Inbox update guard | `lastRenderedInboxDataStr` ref | Prevents unnecessary communication re-renders |
| CSS | Mostly inline `<style dangerouslySetInnerHTML>` | No external CSS file for workstation (fast, no flash) |
| Fonts | Next.js Google Fonts (preloaded) | Geist Sans, Geist Mono |

## Performance Concerns

### KP-001 | N+1 risk in `/api/conversation/shared`
**File**: `src/routes/conversationRoutes.js` lines 212–222  
**Issue**: For each trade in active queues, it calls `conversationEngine.getConversation(tradeRef)` individually — potential N+1 query if many active queues exist.  
**Fix**: Batch all tradeRefs, do a single `Conversation.find({ tradeRef: { $in: refs } })`.

### KP-002 | Full `Trade.find({})` on `/api/trade/all`
**File**: `src/routes/tradeRoutes.js` line 19  
**Issue**: Admin-level endpoint loads ALL trades with no pagination. Could be thousands.  
**Fix**: Add pagination (`skip`/`limit`) or use cursor-based pagination.

### KP-003 | Large communication page (855 lines, 41KB)
**File**: `frontend/src/app/communication/page.js`  
**Issue**: Single massive component — slow to parse and hydrate.  
**Fix**: Split into sub-components; lazy-load message thread.

### KP-004 | In-memory trade cache may grow unbounded
**File**: `server.js` lines 76–89  
**Issue**: `_cachedTrades` is rebuilt from `Trade.find({ assignedTo: { $ne: null } })` every 2 seconds. If 1000s of trades are assigned, this is a heavy periodic query.  
**Fix**: Use TTL cache or targeted cache invalidation instead of full rebuild.

### KP-005 | `JSON.stringify` comparison for trade sync
**File**: `frontend/src/app/workstation/page.js` line 130  
**Issue**: `JSON.stringify(updatedTrade) !== JSON.stringify(selectedTrade)` on every queue update is O(n) string comparison.  
**Fix**: Compare by a few key fields (currentStatus, pendingAmendments.length) instead of full serialization.

## Optimization Targets (Priority Order)

1. **Fix KP-001** (conversation N+1) — medium complexity, measurable DB query reduction
2. **Add pagination to KP-002** — small change, critical for production scale
3. **Fix KP-004** (trade cache strategy) — medium complexity, reduces 2s DB load
4. **Fix KP-005** (JSON comparison) — 5-minute fix, small but repeated improvement
5. **Split KP-003** (communication page) — high effort, improves UX and load time

## Database Index Strategy

Currently indexed fields (from model definitions):
- `Trade.tradeRef` — unique
- `Trade.nextDesk`
- `Trade.assignedTo`
- `Queue.userId` — unique
- `Queue.isActive`
- `Conversation.tradeRef` — unique
- `AuditLog.tradeRef`
- `AuditLog.userId`
- `FOCommunication.tradeRef` — unique
- `UserScore.userId` — unique

**Recommended additions**:
- `Trade.currentStatus` — for filtering by status
- `Trade.isAutoGenerated + Trade.assignedTo` — for pool query in queueComposer
- `Queue.desk + Queue.isActive` — for desk-specific active queue lookups

## Socket.io Performance

- Emits are room-targeted where possible (`user_<userId>`) to avoid global broadcasts
- `new_email` is currently emitted globally (`io.emit`) in several places — should be room-targeted
- Each connected user joins both `user_<userId>` AND `desk_<desk>` rooms — overhead is minimal for expected user counts (<100 concurrent)
