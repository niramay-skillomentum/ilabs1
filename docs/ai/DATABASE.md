## Database Engine

**MongoDB Atlas** (cloud-hosted) via **Mongoose v9.7.0** ODM.

- Connection string: `MONGO_URI` environment variable (Atlas SRV with SSL + replica set)
- Fallback: if `MONGO_URI` is missing or unreachable, the server continues in **memory-only mode** — Mongoose operations degrade gracefully but data is not persisted
- Schema management: Mongoose creates collections automatically on first write (no migration scripts required for new installs)

#### `truths` Sub-document

The core of the simulation's "hidden truth" model. Each desk has its own authoritative source.

```json
{
  "truths": {
    "universal": { "amount": 1000000, "valueDate": "...", "currency": "USD", "counterparty": "CITI" },
    "mo":         { "amount": 1000000, "valueDate": "...", "currency": "USD", "counterparty": "CITI" },
    "confirmation": { "amount": 1050000, "valueDate": "...", "currency": "USD" },
    "settlement": { "amount": 1000000, "valueDate": "...", "currency": "USD", "counterparty": "CITI" }
  }
}
```

- **`universal`**: The absolute correct economics. Used for CPTY concession logic.
- **`mo`**: What the Front Office says. MO break = `truths.mo ≠ booking`.
- **`confirmation`**: What the Counterparty says. Confirmation break = `truths.confirmation ≠ trade economics`. **Note**: No counterparty field — by design.
- **`settlement`**: Settlement-level truth (mirrors universal in current implementation).

#### `booking` Sub-document

What the ops booking system shows to the trainee. May differ from `truths.mo` if a break was injected.

```json
{
  "booking": {
    "amount": 950000,
    "valueDate": "...",
    "currency": "USD",
    "counterparty": "CITI"
  }
}
```

#### `confirmationScenario` Sub-document

Metadata about the CPTY-level dispute, set at trade generation time.

```json
{
  "confirmationScenario": {
    "disputeType": "AMOUNT",
    "expectedEconomics": { "amount": 1050000, "valueDate": "...", "currency": "USD" },
    "evidence": [{ "type": "SWIFT", "provided": false, "requestedAt": null, "receivedAt": null }]
  }
}
```

`disputeType` is `null` for trades with no confirmation break.

#### `foEscalation` Sub-document

Tracks FO escalation from the Confirmation desk.

```json
{
  "foEscalation": {
    "status": "FO_SUPPORTS_US",
    "escalatedAt": "...",
    "resolvedAt": "...",
    "foResponse": "We have reviewed and confirm your booking is correct."
  }
}
```

FO escalation statuses: `null` → `PENDING` → `FO_INVESTIGATING` → `FO_SUPPORTS_US` / `FO_FOUND_ERROR`

#### `conversation` Sub-document

Persists the email conversation resolution state across server restarts.

```json
{
  "conversation": {
    "status": "RESOLVED",
    "resolvedAt": "2026-06-26T10:30:00Z"
  }
}
```

Status values: `null` (no conversation yet), `OPEN`, `RESOLVED`

#### `amendmentHistory` Array

Full audit of all amendments applied across the trade lifecycle.

```json
[{
  "amendmentNumber": 1,
  "desk": "MO",
  "field": "amount",
  "oldValue": 950000,
  "newValue": 1000000,
  "source": "FO_EMAIL",
  "status": "ACCEPTED",
  "appliedAt": "...",
  "appliedBy": "user@example.com"
}]
```

#### Trade Status Enum

| Status | Desk | Meaning |
|--------|------|---------|
| `MO_PENDING` | MO | Trade in queue awaiting MO validation decision |
| `MO_BREAK_OPEN` | MO | Break raised by user; must contact FO before validating |
| `PENDING_FO_RESPONSE` | MO | Email sent to FO; awaiting FO reply |
| `CONFIRMATION_PENDING` | CONFIRMATION | Passed MO; awaiting CPTY confirmation |
| `LIASING_WITH_CPTY` | CONFIRMATION | Email sent to CPTY; awaiting CPTY response |
| `CONFIRMATION_BREAK` | CONFIRMATION | CPTY disputed; formal break open |
| `LIASING_WITH_FO` | CONFIRMATION | FO channel opened for dispute support |
| `SETTLEMENT_PENDING` | SETTLEMENT | Confirmed; awaiting settlement approval |
| `READY_FOR_APPROVAL` | SETTLEMENT | Settlement approved; ready for final sign-off |
| `SETTLEMENT_BREAK` | SETTLEMENT | Settlement fail detected |
| `LIASING_WITH_CPTY` (settle) | SETTLEMENT | Following up with CPTY on settlement fail |

## Query Patterns

### Queue Building (`queueComposer.js`)
```js
// Count available pool for a desk
Trade.countDocuments({ nextDesk: desk, assignedTo: null, currentStatus: { $in: statusList } })

// Fetch clean/break trades from pool
Trade.find({ nextDesk: desk, assignedTo: null, currentStatus: { $in: moStatuses } }).lean().limit(N)

// Assign trades to user
Trade.updateMany({ tradeRef: { $in: refs } }, { $set: { assignedTo: userId } })
```

### Active Queue Fetch
```js
// Get user's active queue
Queue.findOne({ userId, isActive: true })

// Batch-load trade documents
Trade.find({ tradeRef: { $in: queueDoc.trades } }).lean()
```

### Trade Action
```js
// Always re-fetch from DB with ownership check — never trust client
Trade.findOne({ tradeRef, assignedTo: userId })

// Patch specific fields after action
Trade.updateOne({ tradeRef }, { $set: { currentStatus, foResponseReceived, ... } })
```

### Cache Refresh (every 2s)
```js
Trade.find({ assignedTo: { $ne: null } }).lean()
// → rebuilds _cachedTrades object
```
